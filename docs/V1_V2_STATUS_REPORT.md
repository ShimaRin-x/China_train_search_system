# V1/V2 完成情况与 V2 开发计划报告

> 2026-06-12 更新：已新增 OSM/OpenRailwayMap 车站坐标导入器，并完成北京南附近与北京-上海走廊的 `match-only` 导入。当前 G1 的 7 个经停站均已补到经纬度。全量 CN/TW/HK/MO 行政区一次性 Overpass 查询发生 504，后续应按省份或区域分批导入。

## 一、总体结论

当前项目已经完成 V1 的主要工程骨架：前端地图工作台、铁路底图样式、车次搜索面板、车站搜索面板、后端 API、PostGIS 数据模型、Docker Compose 方案、API 文档、ER 图和 `cr-12306-train-info` 离线导入能力都已经具备。

但严格按 V1 MVP 验收标准看，V1 还不能算完全验收通过。主要原因不是界面问题，而是数据链路还没有打通：

- `cr-12306-train-info` 只能提供车次、经停站、到发时间等运行信息，不能提供可靠车站坐标。
- OSM/OpenRailwayMap 的车站坐标、铁路线路几何导入尚未完成。
- 车次经停站到真实铁路网的路径匹配尚未完成，`train_route_segments` 目前没有真实经路数据。

因此当前状态可以判断为：

- V1 界面/API/数据库结构：基本完成。
- V1 真实地图验收：未完成，卡在站点坐标、铁路线路空间数据、车次经路匹配三项。
- V2：只提前完成了少量基础能力，真正的“车站模式”尚未开始。

## 三、V1 数据库完成情况

当前数据库模型比最初 V1 草案更完整，已经包含：

- `stations`
- `station_search_aliases`
- `railway_lines`
- `rail_segments`
- `train_services`
- `train_stops`
- `train_route_segments`

已完成的点：

- 车站、车次、经停站、铁路线路、经路片段表结构已具备。
- 关键字段已支持 PostGIS 几何类型。
- 已有 GiST 空间索引和 trigram 模糊搜索索引设计。
- `train_services.raw_payload` 已保存 `cr-12306-train-info` 原始数据，便于回溯。

仍缺的点：

- `stations.location` 大量为空，需要从 OSM/OpenRailwayMap 合并坐标。
- `rail_segments.geom` 需要导入真实铁路网。
- `train_route_segments.route_geom` 需要通过路径匹配生成。
- V1 草案中的 `running_time` 当前没有直接字段，现有模型是 `day_offset`、`dwell_minutes`、`arrive_time`、`depart_time`。若前端要展示累计运行时间，需要导入时计算或接口动态计算。

## 四、V1 未完成/阻塞项

### 1. OSM/OpenRailwayMap 数据导入

这是 V1 地图自有数据层的基础。当前已有表结构和 API，但缺少实际导入流程。

需要完成：

- 过滤中国大陆、中国台湾、中国香港、中国澳门范围内的铁路数据。
- 导入铁路站点、停靠点、线路、区间几何。
- 标准化铁路类型：高速铁路、快速/城际、普通客运、普通货运、其他。
- 写入 `stations.location`、`railway_lines`、`rail_segments`。

### 2. 12306 车站名与地图车站坐标合并

当前能从 `cr-12306-train-info` 拿到“北京南、上海虹桥”等站名，但这些记录没有坐标。

需要完成：

- 按站名、别名、拼音、电报码匹配 OSM/OpenRailwayMap 车站。
- 匹配成功后补齐 `stations.location`。
- 对重名站、同城多站、旧站名建立人工校验清单。

### 3. 车次经路匹配

V1 验收核心是输入 G1 后地图显示真实运行经路。当前缺少经停站之间的铁路网路径匹配。

需要完成：

- 将铁路网构造成可寻路图。
- 对相邻经停站做路径搜索。
- 按线路类型、距离偏差、方向、线路名等给候选路径打分。
- 写入 `train_route_segments`。
- 低置信度结果保留人工校验标记。

### 4. 经停站 Marker 与弹窗

需要补充：

- 起点绿色 Marker。
- 中间站蓝色 Marker。
- 终点红色 Marker。
- 点击 Marker 显示站名、到达时间、发车时间、累计运行时间。

### 5. 选中车次时其他线路淡化

当前前端已有高亮线路图层，但“其他线路淡化”还没有形成严格的选中态逻辑。后续可以在选中车次时降低自有 `railways-line` 透明度，同时保持底图铁路不抢高亮线路。

## 五、V1 验收状态

## 八、V2 开发计划

### 阶段 0：先补齐 V1 数据基础

V2 依赖真实车站坐标、铁路线路和车次经路，所以建议先完成 V1 数据底座。

任务：

- 实现 OSM/OpenRailwayMap 站点和铁路线路导入。
- 将 `cr-12306-train-info` 车站名与地图车站坐标合并。
- 实现相邻经停站之间的铁路网路径匹配。
- 写入 `train_route_segments`。
- 完成 G1 验收闭环。

产出：

- `stations.location` 有坐标。
- `rail_segments.geom` 有真实线路。
- `train_route_segments.route_geom` 有真实车次经路。
- V1 可以显示经停站 Marker、起终点 Marker、真实经路高亮。

### 阶段 1：车站详情 API

新增 API：

- `GET /api/v1/stations/{station_id}`
- `GET /api/v1/stations/{station_id}/trains?limit=50&offset=0`
- `GET /api/v1/stations/{station_id}/lines`
- `GET /api/v1/stations/{station_id}/routes?limit=20&offset=0`

实现逻辑：

- 经过车次直接从 `train_stops -> train_services` 查询。
- 经过线路从 `rail_segments.from_station_id/to_station_id` 或站点邻近线路空间关系查询。
- 经过车次经路从 `train_route_segments` 查询，并按分页返回。

### 阶段 2：车站模式前端

新增交互：

- 点击地图车站后打开车站详情面板。
- 面板展示经过车次列表。
- 面板展示经过铁路线路列表。
- 支持点击某个车次后复用 V1 的经路高亮逻辑。
- 支持“显示经过本站的车次经路”，但默认分页加载，不一次性渲染全部。

建议界面结构：

- 车站标题区：站名、城市、省份、电报码。
- 标签页：经过车次、经过线路、经路展示。
- 车次列表：车次号、始发终到、本站到发时间。
- 线路列表：线路名、线路类型、方向/相邻站。

### 阶段 3：线路聚合与性能优化

禁止一次渲染 1000 条独立车次线。

建议方案：

- 默认只显示前 20 条经过车次经路。
- “显示更多”分页加载。
- 相同 `rail_segment` 聚合成一条线，用经过车次数控制线宽或颜色深浅。
- 低缩放只显示聚合线路，高缩放再显示具体车次。
- 对车站查询、线路查询、聚合经路结果加 Redis 缓存。

可新增聚合接口：

- `GET /api/v1/stations/{station_id}/route-aggregate`

返回：

- 线路 GeoJSON。
- 每段线路经过车次数。
- 代表车次列表。

### 阶段 4：统一搜索

新增统一搜索 API：

- `GET /api/v1/search?q=南京南&types=train,station,line&limit=20`

返回分组：

- `trains`
- `stations`
- `lines`

前端改造：

- 顶部保留一个统一搜索入口。
- 搜索结果按类型分组展示。
- 点击车次进入 V1 车次经路模式。
- 点击车站进入 V2 车站模式。
- 点击线路进入线路详情模式。

### 阶段 5：V2 验收

以 `南京南` 为验收样例：

- 点击或搜索南京南。
- 能看到经过车次列表。
- 能看到经过线路列表。
- 选择车次后能显示该车次经路。
- 选择“显示经过本站车次经路”时，地图不会卡顿，默认分页或聚合显示。

## 九、建议优先级

1. 先补 V1 数据导入和坐标合并。
2. 再做车次经路匹配，确保 G1 能真实验收。
3. 补经停站 Marker 颜色和 popup。
4. 再进入 V2 的车站反查车次/线路。
5. 最后做统一搜索和聚合性能优化。

当前最关键的下一步不是继续改前端样式，而是把车站坐标、铁路线路、车次经路三类数据串起来。
